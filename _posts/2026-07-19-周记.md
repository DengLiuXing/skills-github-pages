# 本周探索记录：个人知识库搭建与跨系统开发环境配置
本周主要精力集中在横向的技术探索上，完成了个人博客的初步搭建以及基于 Linux 的仿真开发环境的配置。以下是具体的实践记录与思考：
## 1. 基于 Jekyll 的静态博客搭建
本周在 GitHub 上基于 Jekyll 完成了静态私人博客界面 [DLX的个人博客](https://dengliuxing.github.io/skills-github-pages/) 的开发，作为个人技术探索的第一个落脚点。
*   **当前优势：** 尽管与具体的编程核心技术关联不大，且 Jekyll 框架本身已较为陈旧，缺乏对现代前端特性的友好支持，但其有**极简的内容管理方式**。仅需编写 Markdown 文件即可完成文档编辑，非常适合暂用作个人的日记本和资料库。
*   **技术拓展：** 在搭建过程中，我顺带粗略了解并体验了 **GitHub Actions** 的自动化构建与部署功能。
*   **未来规划：** 考虑到 Jekyll 的局限性，后期我计划基于 **Python** 或 **Go** 语言，将博客重构升级为一个更具现代特性、更易个性化的动态/静态网站。
## 2. WSL 环境配置与 Gazebo 仿真器安装
本周另一项重要工作是在 Windows 中配置 WSL 环境，并成功安装了 Gazebo 仿真器。
*   **为何选择 WSL**：相比传统的虚拟机方案，WSL 是一种较新的多系统共存方案。它直接在 Windows 中使用真实的 Linux 内核，具备两大显著优势：
    *   **更轻量**：资源占用远低于传统虚拟机。
    *   **弱隔离的便利性**：虽然隔离性弱，但在配置日常开发环境时，这种弱隔离使网络、硬件和 Windows 系统的资源基本与 Linux 环境直接互通，反而能带来极大的便利，（且通常我们很少在开发环境中处理高风险文件，似乎并没有使用笨重的虚拟机的必要了）。
*   **踩坑与解决过程**：在配置 Gazebo 的过程中，我经历了一段典型的“追新”到“求稳”的探索过程：
    *   **初次尝试（盲目追新）**：因个人偏好新事物，一开始配置了今年刚出的 Ubuntu 26.04。结果发现 Gazebo 尚未提供支持该最新版本的安装包，方案受阻。
    *   **二次尝试（极度求稳）**：转变策略，开了完全稳妥的 Ubuntu 22.04 环境，并且尝试安装较为成熟的 Gazebo Fortress 版本。结果发现一旦进行图像渲染就会触发崩溃。
    *   **最终成功（折中方案）**：放弃 Fortress，转而尝试较新的 Gazebo Garden 版本，终于成功运行。
*   **经验与反思**：针对 Fortress 版本在 WSL 2 中的渲染崩溃问题，我查阅资料后得出了一些技术背景分析：
    *   **生态时间线错位**：WSL 2 的生态相对年轻，直到 2022 年才发展壮大并逐渐步入主流；而 Gazebo Fortress 发布于 2021 年。
    *   **根本原因推测**：Fortress 在开发时大概率未曾考虑过对 WSL 2 的底层优化与兼容，从而在后续适配中导致了各种奇怪的依赖与渲染问题。
*   **收获**：通过这一系列的排错与探索，我不仅成功搭建了以后可能会需要用的虚拟仿真环境，还借此契机较为系统地了解了 **Linux（Debian 系）的基本特性与日常使用方法**，为后续更深度的开发打下了基础。
